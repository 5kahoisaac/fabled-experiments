Result

Fabled reliably bolts good structure onto a model’s output, but it cannot install reasoning a model lacks — so it rescues weak models and adds only overhead to the strongest one.

The Outcome column in the table above is each author’s own self-report. The chart and analysis below use an independent browser-verified evaluation (every build driven live under software WebGL / SwiftShader), which re-scored several runs the authors had logged as “failed” or “unverified” as actually playable. Where the two disagree, the browser-verified result is authoritative — see caveats.

Four-quadrant comparison

Did Fabled really improve the model’s thinking?

Verdict: Fabled improves the OUTPUT, not reliably the REASONING — it scaffolds the shape of good engineering thinking without supplying the thinking itself. Across the seven comparable fabled-vs-control pairs (same model, same harness), Fabled helped 5 and hurt 2 — and both regressions land on the strongest base model, gpt-5.5.

The capability-equalizer thesis. The cleanest proof that Fabled transplants structure rather than teaches reasoning is that the strongest models already produce that structure without it. The fable-5 baseline (skill = none) independently split a pure logic block from the render layer “specifically because it foresaw verification,” shipped 36 logic assertions, pinned Three.js, and guarded CDN/WebGL/audio failures behind a friendly overlay — every habit Fabled would otherwise impose. The gpt-5.5/Codex control did the same on its own initiative. When the discipline is endogenous, the skill has nothing to add and its process overhead can only cost points — which is exactly why the two regressions are both gpt-5.5.

Honest framing: the −4.0 and −2.0 deltas are within single-rater eval noise — read them as “no benefit on a model that didn’t need it,” not demonstrated harm. The truth is a gradient: Fabled reliably improves output structure, weakly and indirectly supports reasoning through that structure, and does not supply reasoning depth a model lacks.

The decisive limit — gemma still mis-reasoned with the full scaffold. gemma-4 followed Fabled’s form faithfully (plan, assumptions, contracts, logic/render split) yet committed the same core error three times running: a vertical-axis sign-convention bug in the flight physics that it “fixed” by flipping signs and tweaking the magnitudes the user named, rather than re-deriving the physics from first principles — converging only on the 6th iteration. qwen3.6 is the matching anti-pattern: it produced the Plan, the Gate-B scan and a Verification table all marked PASS, but substituted Fabled’s executable check with a lexical string scan and never ran the game — shipping an unplayable build whose one-line boundary invariant was assertable but never asserted. Fabled’s form was followed; its function was not. Where the model is too weak to diagnose from first principles, the gates get satisfied ceremonially while the reasoning failure ships untouched.

Why the small models diverge (gpt-5-mini vs gemma-4 vs qwen3.6)

They diverge not because Fabled adds quality uniformly, but because each model’s competence bottleneck sits in a different place, and Fabled only pays off when the bottleneck is the exact thing it scaffolds.

gpt-5-mini — the rescue (bottleneck: dependency plumbing). The standout swing, 50.0 → 75.6. The control’s failure axis was pure delivery fragility: an import-map plus six live CDN example-module imports that never resolved (“Failed to resolve module specifier ‘three’” + cascading 404/500s). The author chased symptoms across eight prompts and a 2.07M-token spiral, then aborted. Fabled fixed this structurally — one pinned UMD bundle, zero example-module imports, a real WebGL fallback — shipping clean in two prompts and 78k tokens. Fabled’s single-file + pinned-CDN + fallback discipline is the precise fix for a dependency-plumbing failure, so its leverage was maximal.

gemma-4 — persistent control-inversion (bottleneck: numeric/geometric reasoning). The scaffold ran faithfully and didn’t help where it mattered: a physics sign-convention error a checklist cannot encode, surviving three correction rounds because the model “kept tweaking magnitudes instead of re-deriving from first principles.” Tellingly, gemma’s control never hit this trap — its steering turns were operational (a foreground vite hang, a clobbered render()). (Caution: attributing gemma’s harder path to “Fabled dragging it into a full-physics build” is a causal inference on a single open-weights run.)

qwen3.6 — ceremony displaces verification. qwen’s control got the physics right on the first shot (playable in 2 prompts / 139k tokens). Its fabled run took 3 prompts / 506k tokens and was logged a process failure — net-negative on its own efficiency — because the ceremony displaced the cheap executable check that would have caught a spawn-on-the-boundary collision bug. Score and process diverge here: the evaluator scored the eventually-repaired artifact 65.6 vs 54.6, so “Fabled helped qwen” is true on the rubric but false on the run’s lived outcome.

The quantization angle — a hypothesis, not a finding. Both quantized local models shipped a physically-wrong core (4-bit qwen: boundary collision; 8-bit gemma: inverted sign) that hosted full-precision gpt-5-mini never did. Generation was never the bottleneck for any of them — all three authored a pretty Three.js scene in one write — the bottleneck was reasoning about correctness, where lower precision appears to show. But this is a 3-point local-vs-hosted contrast confounded by architecture, harness and training; treat it as a hypothesis to test, not a result.

Fun fact: gpt-5.5 via Pi beats gpt-5.5 via Codex CLI

The same model is dramatically leaner under the Pi harness than under Codex CLI — and the gap is almost entirely the harness, not the thinking. For the like-for-like pair (stock Pi vs stock no-OMO Codex), Pi used ~3.8× fewer tokens (287k vs 1.10M), ran ~2.1× faster (210s vs 449s), and cost ~4.1× less ($0.73 vs ~$3.00) — at statistically indistinguishable quality (both 68.6).

Mechanism: re-fed context, not extra work. The output tokens are tiny and similar everywhere — Pi ~8.2k, Codex no-OMO 15.5k, Codex-fabled 25.2k. The blow-up is the envelope of context re-read around that small core. Pi commits to one architecture, writes the whole game in a single shot, and closes with one top-level browser check — few tool round-trips, little re-ingested context. Codex CLI runs a long inspect → scaffold → serve → verify → tune loop, and each round-trip re-sends the accumulated, growing context; the fingerprint is the rising cache-read share (~71% of Pi’s total vs ~86% for Codex no-OMO). (This is reconstructed from cache-share trends and loop shapes, not per-tool-call telemetry, and token accounting isn’t strictly comparable across harnesses — lean on the within-family direction, not exact ratios.)

The nuance that keeps it honest: Codex CAN win on peak quality — but only by paying enormously. The single highest challenger score, 84.2, was Codex no-OMO + Fabled — and it cost ~2.76M tokens, ~1068s and ~$4.82, roughly 16× Pi’s tokens and ~5× its wall-clock. That peak wasn’t the harness being smarter; it was Fabled’s executed logic tests + dependency-fallback gates surviving Codex’s long-loop transport. Crucially Fabled did not transfer that lift to Pi (Pi-fabled 64.6, just below stock Pi). So: Pi wins decisively on efficiency at equal quality; Codex + Fabled wins on peak quality only by spending a fortune. And the harness has a tail risk — glm-5.1 under the Codex-family OpenCode harness died on ~60-second stream deaths and shipped nothing (the lone true FAIL), a stability cost that never shows on the happy path.

How the quadrants land

The sweet spot is the bottom-left: low cost, low time, small green bubble — a PASS earned without burning tokens or wall-clock. The standout is gpt-5-mini-gh-pi-fabled, effectively the origin of the chart: cheapest ($0.034), fastest (~127s), tiniest footprint (78k tokens), and a PASS (75.6) — a weak, cheap model that, given the right scaffold for its bottleneck, beats far more expensive runs on the only axis that’s free. The expensive outliers cluster top-right: the four Codex runs are the giant bubbles in the costly corner — they show that throwing tokens and time at the problem buys peak quality at the extreme (84.2) but mostly buys amber (66–68). The local-MLX runs sit at zero cost but drift up the time axis with deceptively huge bubbles — a reminder their token totals are full-prompt re-counts, not build size.

Methodology & caveats

n = 1 taste. Each cell is a single run scored by a single evaluator against subjective targets (“pretty,” “playable”). Score deltas inside roughly ±5 points — including both gpt-5.5 regressions — are within noise; treat them as “no benefit,” not measured harm. Directional and mechanical claims (read off shipped code: import failures, sign conventions, unasserted invariants) are far firmer than absolute magnitudes.

Token accounting is NOT comparable across harnesses. Local MLX models (qwen, gemma) re-meter the full prompt every turn with no cache-read field, so their 506k/843k totals are prefix re-counts, not new-token work. Pi and GH report cache-read separately; OpenCode logged nothing; the fable-5 baseline totals are explicitly estimated. Cross-harness token/cost ratios cannot bear weight — only same-harness within-pair comparisons are legitimate, and even those reflect steering and cache, not build size.

Browser-verification overturned several self-reports. Multiple runs the authors logged as “failed/unverified” were re-scored PASS or PASS-WITH-DEFECTS by the independent evaluator under software WebGL. The authoritative scores are the browser-verified ones; only glm-5-1-fabled-oc truly produced nothing — a transport-layer (~60s stream-death) FAIL, not a reasoning failure.

Two cells were scored pre-SwiftShader. The fable-5 baseline (84.6) and gemma-4-pi-fabled (57.6) were evaluated before the software-WebGL render path, where scenes that didn’t composite may have been penalised; both are likely mildly understated. Don’t over-read small gaps involving them.

“Reasoning” is partly inferred. The thinking-process characterisations come from post-hoc JOURNEY narrations and shipped artifacts, not raw chain-of-thought — so causal verbs (“Fabled dragged the model into a harder build,” “quantization causes the failure”) are inferences on single open-weights runs, not controlled findings.

Takeaway — Fabled is a capability equalizer that scaffolds output structure for models that can’t generate it themselves — rescuing weak/cheap models (gpt-5-mini: a 2.07M-token spiral → a 78k-token clean PASS) while adding only overhead to the strongest base model — but it installs the shape of good engineering, not the reasoning, so where a model can’t diagnose from first principles the gates pass ceremonially and the real bug ships anyway.
